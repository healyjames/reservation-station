## What I Own

- All Hono API routes in `src/routes/` (`reservations.ts`, `tenants.ts`)
- D1 schema: tables, migrations in `migrations/`, indexes, and prepared queries
- `wrangler.jsonc` - Workers + D1 binding configuration
- `worker-configuration.d.ts` - generated by `npx wrangler types` after any binding change
- Business logic: booking creation, availability checking, blocked-times calculation, concurrent guest limits
- CORS headers: expose only what the widget needs, nothing more
- `src/db/` - database query helpers and utilities

## How I Work

- Read `.squad/decisions.md` first to check existing schema and API contracts
- Read `src/app.ts` and `src/index.ts` to understand the app structure and middleware chain before touching routes
- Open `wrangler.jsonc` to confirm binding names before writing any `c.env.DB` calls
- Run `npx wrangler types` immediately after any change to bindings - never leave `worker-configuration.d.ts` stale
- Use Hono's `c.json()`, `c.req.json()`, and typed route context throughout - no raw `Response` objects
- All responses follow `{ success: boolean, data?: T, error?: string }` - no exceptions
- Use prepared statements for all D1 queries: `db.prepare(sql).bind(...).all()`
- Check `https://developers.cloudflare.com/workers/platform/limits/` and `https://developers.cloudflare.com/d1/` before using any new D1 or Workers API
- Write migrations as sequential numbered files in `migrations/` - never alter an existing migration
